02 — Componentes

As 5 pecas do sistema

Cada peca resolve um problema diferente. Juntas, criam um ciclo de desenvolvimento que roda quase sozinho.

⚙

Agent Teams

Grupos de 2-4 agentes especializados trabalhando em paralelo. Cada um recebe um prompt com escopo claro, edita arquivos, roda testes e reporta ao lead. Resultado: 3x mais rapido que sequencial.

Paralelo
↺

Cron Loops

Jobs automaticos a cada 5 minutos: rodar testes, monitorar PRs, processar mensagens Telegram. Detectam problemas antes do usuario perceber. Zero intervencao humana.

Automatico
✉

Telegram

Canal bidirecional: usuario manda feedback do celular, agente categoriza (bug, feature, status, pergunta) e age automaticamente — cria issues no GitHub, roda testes, responde.

Tempo real
👁

Chrome DevTools

Conexao direta com o Chrome para auditar a UI visualmente: tirar screenshots, inspecionar elementos, verificar console, testar no mobile. Encontra bugs que testes unitarios nao pegam.

Visual
⟳

Ralph Loop

Loop de iteracao: roda testes, se falhar le o erro, corrige o codigo, roda de novo. Repete ate verde ou 5 tentativas. Garante que o codigo sempre termina funcional.

Resiliente
03 — O ciclo

Como cada fase funciona

Toda fase segue o mesmo ciclo de 6 etapas. Previsivel, repetivel, confiavel.

1. Planejar

Define o escopo no CLAUDE.md. Cria tasks com descricoes detalhadas. Lista arquivos que serao tocados.

2. Executar

Cria team, spawna teammates com prompts especificos. Cada agente trabalha independente, editando seus arquivos.

3. Integrar

Lead recebe resultados, verifica conflitos, corrige funcoes faltando, fixa escaped characters. O ponto mais critico.

4. Verificar

Roda npm test (58 testes), node -c app.js (syntax), Chrome DevTools (visual). Tripla verificacao.

5. Entregar

Commit convencional, push pro GitHub, notifica usuario no Telegram. Shutdown teammates, delete team.

6. Monitorar

Crons continuam rodando: testes a cada 5min, PRs monitoradas, Telegram respondido. Mesmo entre fases.

04 — Agent Teams em detalhe

Trabalho paralelo na pratica

Exemplo real: Fase 8 criou 6 novos endpoints + UI + testes com 3 agentes simultaneos.

// Team Lead cria o time
TeamCreate({ name: "phase8-features" })

// Spawna 3 agentes em paralelo
Agent({ name: "api-expand", prompt: "Cria 6 endpoints..." })
Agent({ name: "feature-ui", prompt: "ESPERA api-expand, cria UI..." })
Agent({ name: "feature-test", prompt: "ESPERA ambos, escreve testes..." })

// Quando terminam:
// 1. Lead verifica testes
// 2. Corrige integracoes
// 3. Commit + push
// 4. Shutdown teammates

Padroes de coordenacao

Pipeline

A → B → C

Backend termina, frontend comeca, tester depois. Dependencia sequencial.

Paralelo

A | B | C

Todos trabalham ao mesmo tempo. Maximo throughput, risco de conflito.

Hub-spoke

Lead ↔ A, B, C

Lead distribui, recolhe e integra. Mais seguro, mais lento.

07 — Licoes aprendidas

O que deu errado e como resolvemos

01

Teammates editando o mesmo arquivo

Multiplos agentes escrevendo no app.js causou funcoes removidas por um que outro chamava. Solucao: testes de integracao que verificam sintaxe JS e existencia de funcoes.

02

Endpoints TMDB inexistentes

Um teammate criou /tv/upcoming que nao existe na API do TMDB. Crashou a pagina de series. Solucao: validar endpoints contra documentacao antes de implementar.

03

Rotas Express com wildcard

/api/tv/:id matchava antes de /api/tv/search porque Express processa na ordem de registro. Solucao: rotas especificas SEMPRE antes de wildcards.

04

Caracteres escapados (\!)

Agentes rodando em zsh escapavam ! como \! no JavaScript, quebrando o codigo. Solucao: teste automatico que detecta \! e replace_all apos cada edicao.

05

Linter revertendo edits

Um formatter revertia alteracoes entre edit e commit. Solucao: editar e commitar imediatamente, verificar se a mudanca persistiu.
